1. Introduction & Project Overview

Wanderlust is a full-stack web application inspired by Airbnb that enables users to browse, create, edit, and manage property listings for travel accommodations worldwide.

The Wanderlust project (also branded as "Explore" in the UI) is a comprehensive full-stack web application that replicates the core functionality of modern property listing platforms like Airbnb. It provides a rich, interactive user experience for discovering travel stays across the globe, complete with user authentication, CRUD operations on property listings, and a review system.

The application follows a client-server architecture with a clear separation between the frontend (React.js single-page application) and the backend (Express.js REST API). Data is persisted in a MongoDB database, making this a complete MERN Stack application — one of the most popular and industry-standard technology stacks for modern web development.

📌 Project Scope
The application ships with 28 pre-seeded sample listings spanning locations across the USA, Europe, Asia, Africa, and Oceania — allowing immediate demonstration of all features without manual data entry.

2. Objectives of the Project

The primary objectives of the Wanderlust project are as follows:

  1. Build a Full-Stack Application — Demonstrate proficiency in developing both client-side and server-side components of a modern web application using the MERN stack.
  2. Implement RESTful API Design — Design and implement a well-structured REST API following industry best practices for CRUD operations on property listings.
  3. User Authentication & Authorization — Implement a secure authentication system using Passport.js with session-based authentication and MongoDB session store.
  4. Responsive & Modern UI — Create an aesthetically pleasing, responsive user interface inspired by Airbnb's design language, using React.js with custom CSS styling.
  5. Database Design — Design and implement efficient MongoDB schemas with proper relationships (references) between collections (Listings, Reviews, Users).
  6. Input Validation — Implement robust server-side validation using Joi schemas to ensure data integrity.
  7. Image Handling — Support image uploads via drag-and-drop with Base64 encoding and URL-based image references.
  8. Search & Filter Functionality — Implement client-side search and price-range filtering to help users discover relevant listings.

4.1 Frontend Layer (Client)

The frontend is a React.js Single Page Application (SPA) built with Vite as the build tool. It runs on port 5173 during development and communicates with the backend via HTTP requests using Axios and the native Fetch API. Client-side routing is handled by React Router v7, enabling seamless navigation without full page reloads.

4.2 Backend Layer (Server)

The backend is an Express.js REST API running on port 8080. It handles all business logic including CRUD operations on listings, user authentication (via Passport.js), input validation (via Joi), and session management (via express-session with MongoDB store). CORS is configured to allow requests from the frontend origin.

4.3 Data Layer (Database)

MongoDB serves as the database, running on the default port 27017. The database is named wanderlust and contains three main collections: Listings, Reviews, and Users. Mongoose is used as the ODM (Object Data Modeling) library, providing schema definitions, validation, and query building.

6. Backend Implementation

The backend is built with Express.js 5 and serves as a RESTful API that the React frontend communicates with over HTTP.

6.1 Server Configuration (app.js)

The main server file configures the Express application with the following middleware stack:

// Core Middleware Stack
const express = require("express");
const app = express();

// Body parsing
app.use(express.urlencoded({ extended: true, limit: "10mb" }));
app.use(express.json({ limit: "10mb" }));

// CORS - Allow frontend at localhost:5173
app.use(cors({
    origin: "http://localhost:5173",
    credentials: true,
    methods: ["GET", "POST", "PUT", "DELETE"],
}));

// Session with MongoDB Store (7-day expiry)
app.use(session({
    store: new MongoStore({ mongoUrl: MONGO_URL }),
    secret: "thisshouldbeabettersecret",
    cookie: { maxAge: 1000 * 60 * 60 * 24 * 7 }
}));

// Passport Authentication
app.use(passport.initialize());
app.use(passport.session());

// Server listens on port 8080
app.listen(8080);

6.3 Error Handling

The application uses a custom ExpressError class that extends the native JavaScript Error class with an additional statusCode property. The global error handler catches all errors and responds with the appropriate HTTP status code and message. 

class ExpressError extends Error {
    constructor(statusCode, message) {
        super();
        this.statusCode = statusCode;
        this.message = message;
    }
}

9.2 Session Management

  • Session Store: MongoDB via connect-mongo — sessions persist across server restarts
  • Session Expiry: 7 days (configurable via maxAge)
  • Cookie Settings: httpOnly: true, sameSite: "lax" — prevents XSS access to cookies
  • Lazy Updates: touchAfter: 24 * 3600 — reduces unnecessary session writes

9.3 Password Security

Passwords are never stored in plaintext. The passport-local-mongoose plugin automatically handles password hashing with PBKDF2 algorithm, generating a unique salt for each user. The plugin adds hash and salt fields to the User model and provides authenticate(), register(), serializeUser(), and deserializeUser() methods.

11. Pages & Components

11.1 Navbar Component

A sticky navigation bar (z-index: 200, height: 70px) that appears on all pages. Features a brand logo with "Explore" text, a center search bar that redirects to /listings?search=..., and conditional auth links (Sign Up / Log In when not authenticated, Welcome message + Logout when authenticated).

11.2 Listings Component (Main Grid)

The heart of the application — a 273-line component that displays all property listings in a responsive grid layout. Key sub-features include:

  • Category Bar: 14 Airbnb-style category buttons (Trending, Rooms, Mountains, Castles, Beachfront, etc.) with custom SVG icons, horizontally scrollable and sticky below the navbar.
  • Tax Toggle: A pill-shaped switch to display prices with or without 18% GST.
  • Filter Sidebar: Modal overlay with dual price range sliders (min/max), displaying prices in ₹ (Indian Rupees).
  • Star Ratings: Deterministic fake ratings (3.5–5.0) generated from listing ID for visual appeal.
  • Listing Cards: Image, title, star rating, location, and price per night — each linking to the detail page.
  • Skeleton Loading: Shimmer-animated placeholder cards while data loads.

11.3 ShowListing Page

Displays detailed information for a single listing including a hero image (16:9 ratio), title, location with pin icon, description, and price formatted in ₹ (Indian locale). Includes Edit and Delete action buttons, with delete triggering an animated confirmation modal.

11.4 CreateListing & EditListing Pages

Form pages for creating and editing listings. Both feature the ImageUpload component for drag-and-drop image handling, with a fallback URL input field. The edit page pre-populates form fields by fetching existing data from the API.

11.5 Login & Register Pages

Authentication forms styled as centered white cards on a gray background. The login page includes "Remember me" functionality with show/hide password toggle. The register page includes password confirmation with client-side validation. Both use the native fetch API and store user data in localStorage.

11.6 ImageUpload Component

A reusable drag-and-drop image upload component that reads files as Base64 data URLs via FileReader. Supports drag-over visual feedback, click-to-browse, image preview with hover overlay, and a remove button. Used in both Create and Edit listing forms.

13. UI/UX Design

The application's design is heavily inspired by Airbnb's modern design language, featuring clean typography, spacious layouts, and attention to micro-interactions.

13.1 Design Principles

14. Testing & Validation

14.1 Server-Side Validation

All API inputs are validated using Joi schemas before processing. The validateListing and validateReview middleware functions check incoming request bodies against predefined schemas and return 400 Bad Request errors with detailed messages when validation fails.

14.2 Client-Side Validation

  • Required Fields: HTML5 required attributes on form inputs
  • Password Match: Registration form validates that password and confirm password match before submission
  • Price Validation: Minimum value of 0 enforced via Joi on the server

14.3 Error Handling

  • API Errors: Custom ExpressError class with status codes; global error middleware catches all unhandled errors
  • Async Errors: wrapAsync utility catches promise rejections in async route handlers
  • Frontend Errors: try-catch blocks in API calls with error state displayed to users
  • 404 Handling: Individual routes return 404 for non-existent resources
